The Architecture of Failure-Driven Design

Standard AI coding assistants optimize for "happy paths". This system does the opposite: it optimizes for catastrophe.

1. Knowledge Ingestion (RAG)

The system maintains a vectorized memory of historical cloud incidents (Post-Mortems, ADRs). When a user submits an architecture, we first query a FAISS index to find "semantically similar failure scenarios".

query = "S3 bucket with Lambda"
result = retrieve(query)
>> "AWS S3 Outage 2017 (US-EAST-1)"
>> "DynamoDB Hot Partition Pattern"

2. Deterministic Failure Agents

We do not rely solely on LLM hallucination. We use a hybrid approach where an Agent uses a "Pattern Library" (YAML) to deterministically check for violations.

  • Retry Storms: Checks for synchronous calls without backoff/circuit breakers.
  • SPOF: Checks for single-region or single-instance dependencies.
  • Resource Exhaustion: Checks for unbuffered writes to limited capacity resources.

3. Agentic Architecture

Once failures are identified, the Architecture Agent consults the Rule Engine.

If Failure == RetryStorm and Context == Synchronous, then Mitigation = AsyncQueue (SQS).

This chain of reasoning is strictly defined in the `rules/` module, ensuring that the proposed solution is always a valid AWS pattern.
